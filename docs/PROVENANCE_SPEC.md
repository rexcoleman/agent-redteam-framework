# PROVENANCE SPEC

<!-- version: 1.0 -->
<!-- created: 2026-03-16 -->

## Purpose

This document defines the provenance artifacts generated by each pipeline run
and stored in `outputs/provenance/`. Provenance enables reproducibility audits
by capturing the exact software environment, code state, and resolved
configuration that produced a given set of results.

## Artifact Inventory

| File | Contents | Generated By |
|------|----------|-------------|
| `versions.txt` | Pinned versions of every installed Python package (`pip freeze` output) | `reproduce.sh` |
| `git_info.txt` | Git commit hash, branch, date, author, and subject of HEAD | `reproduce.sh` |
| `config_resolved.yaml` | Fully resolved experiment configuration (seeds, agent definitions, LLM settings, attack/defense parameters) | `reproduce.sh` |

## File Specifications

### versions.txt

- Format: `package==version`, one per line (pip freeze format).
- Captures the complete conda/pip environment at execution time.
- Used to detect dependency drift between runs.

### git_info.txt

- Fields: `commit`, `branch`, `date`, `author`, `subject`, plus a
  `git log (last 10)` section for recent history context.
- The commit hash is the primary reproducibility anchor; combine with
  `config_resolved.yaml` to fully reconstruct any experiment.

### config_resolved.yaml

- Contains all parameters that influence results: random seeds, agent
  configurations (framework, type, tools, system prompts, max iterations),
  multi-agent crew definitions, LLM provider/model/temperature settings,
  and attack scenario references.
- Serves as the single source of truth for what was actually run, as opposed
  to what the code defaults might suggest.

## Generation Protocol

1. `reproduce.sh` generates all three files at the start of every full
   pipeline execution, **before** any attack or defense evaluation begins.
2. Files are overwritten on each run; prior versions are preserved only in git
   history.
3. If any provenance file fails to generate, the pipeline should halt.

## Verification Protocol

To verify that current outputs match their provenance:

1. **Environment match**: `diff <(pip freeze) outputs/provenance/versions.txt`
   should produce no differences (or only expected build-path differences).
2. **Code match**: `git rev-parse HEAD` should equal the `commit` field in
   `git_info.txt`.
3. **Config match**: Compare `outputs/provenance/config_resolved.yaml` against
   `config/base.yaml` to confirm no untracked overrides.

## Downstream Consumers

- `tests/test_results.py` — validates output integrity against provenance.
- `docs/REPRODUCIBILITY_SPEC.md` — references provenance as the reproducibility anchor.
- `FINDINGS.md` — cites provenance commit hash for result traceability.

## Retention Policy

Provenance files are committed to version control alongside the outputs they
describe. Each tagged release must include a matching provenance snapshot.
